DHQ encoding documentation

This document provides an introduction to using the DHQ markup language. Because DHQ is a customization of TEI, most of the elements in DHQ markup are fully documented as part of the TEI Guidelines. This guide is intended to provide an overview of the DHQ document structure and information about good practice; it also documents the DHQ-specific elements and usages that may depart from standard TEI usage.

Checklist for Encoding

The following is a checklist of things to look for in the encoded document.

If converting via OxGarage, you may need to do significant cleanup of the resulting TEI. The following are some possible conversion artifacts that should be addressed, and there may be others not listed here:

  • <p type="endnote text"> or <p type="footnote text"> around the content of notes
  • convert all <hi rend="bold">, etc., to semantic markup
  • remove <hi rend="endnote reference"> (or "annotation reference", "footnote reference" etc. as appropriate)
  • remove @place from <note>
  • remove unnecessary attributes from <graphic> (anything except @url)
  • remove <desc> inside <figure> once no longer needed

In addition, the following features should be encoded:

  • All titles should be encoded as <title>; use @rend to distinguish between titles rendered in italics, in quotation marks, or with no rendition
  • Any technical terms (or words which are treated as special terms, even if not strictly technical) should be encoded with <term>
  • Any foreign-language words should be encoded with <foreign>
  • Any words or phrases that have attention drawn to them for other reasons (e.g. ironic distance, words being discussed as words) should be encoded with <q>
  • Any words or phrases that are deliberately given rhetorical emphasis should be encoded with <emph>
  • Any inline segments of computer code should be encoded with <code>
  • Any block-level chunks of computer code should be encoded with <eg>
  • Any references to XML elements, attributes, or values should be encoded with <gi>, <att>, or <val>
  • Any links to online resources should be encoded with <ref>. The content of <ref> may be a text phrase, in cases where the URL is obvious (for instance, New York Times or Wikipedia) or very long. In cases where the URL is not obvious or where the link is to a specific page, the URL should be given in the content of <ref>. In all cases, the URL should also be given in the content of @target.
  • Any inline citations should be encoded with <ptr>. The @target attribute points to the @xml:id of the relevant <bibl> element in the bibliography.
  • Any quotations should be encoded with <quote>. The @rend attribute indicates whether the quote is inline or block. If the quote is accompanied by a citation, the quote and the citation are enclosed together within <cit>. The <quote> element should have a @source attribute that points to the appropriate <bibl> element.
  • Any footnotes or endnotes should be encoded at the point of attachment with <note>. Any bibliographic references within the notes should be moved into the bibliography; notes should be used only for prose comments. Any note which consists solely of a bibliographic reference can be eliminated.
  • Any lists should be encoded with <list> and <item>. By default, lists are considered to be unordered, but the @type attribute can be used to indicate that a list is ordered (i.e. numbered).

Ask yourself the following questions before considering the encoding complete:

  • Is your DHQ-TEI document valid?
    • Is the entire document structure accurately represented in the nesting of <div> elements?
    • Are all headings encoded with <head>?
    • Have all italicized words been accurately captured with appropriate elements?
  • Is the header metadata complete (i.e. Author Bio, Abstract, Volume, Issue...)
  • Did you apply DHQ Transformation Scenario to Preview article's HTML view?
  • Did you commit your changes from working copy to repository?
  • Did you move the item into the TOC internal preview?

DHQ-TEI Element Descriptions

DHQauthor markup is fairly intuitive, particularly if you are already familiar with the TEI Guidelines. A simple DHQ document consists of a <teiHeader> containing document metadata, followed by a <text> that contains the content of the article. The text can be subdivided into sections represented as <div> elements, and each of these is composed of components such as paragraphs, lists, figures, and the like. Within paragraph-level components, there are smaller elements available to encode names, emphasis, quotations, bibliographic references, foreign-language words, terminology, and other aspects of the verbal texture. At the end of the article there are sections for encoding works cited. As a shortcut, we provide a template with all required metadata elements and guidance on encoding.

Structural Elements

<text>

The main content of a DHQ article is contained within the <text> element. Within <text>, there are three major components. The <front> contains a teaser and abstract, both of which are supplied by the author (either upon submission or after the article has been accepted for publication). The <body> element contains the article proper, including any appendices. The <back> element contains the bibliography of works cited, encoded as <listBibl>.

<body>

In a short article with no internal subdivisions, the body element may simply contain paragraphs and paragraph-level things. In longer articles containing multiple sections, the individual sections are encoded with <div>. The <div> element should not be used if there are no internal subdivisions.

  • Appendices should be encoded with <div>, with a heading of "Appendix".
  • Acknowledgements should be encoded as the final <div> within <body>, with a heading of "Acknowledgements".

<floatingText>

To embed a complete embedded text within a DHQ article (for instance, a letter, a game, a poem that does not originate outside the text), use the <floatingText> element. Its structure is essentially the same as that of <text>. The <floatingText> element may go anywhere within <body> or <div>.

<div>

The <div> element is used to represent subdivisions of the text. This element may nest inside itself as necessary to represent nesting subdivisions. A division should begin with a <head> and includes one or more paragraphs and other paragraph-level structures (e.g. quotes, lists, figures, examples), or further subdivisions. DHQ does not use any @type or @n attributes on <div>. It is also unnecessary to assign IDs to <div> elements unless they are the target of a cross-reference.

Division headings should not consist simply of a number, but should give the reader some idea of the section's topical focus and where the argument is going.

Divisions may also include epigraphs (short quotation with accompanying citation) following the division heading.

  • Appendices should be encoded with <div>, with a heading of "Appendix".
  • Acknowledgements should be encoded as the final <div> within <body>, with a heading of "Acknowledgements".

<head>

The <head> element is used for all headings (of articles, sections, tables, figures, examples, etc.). The <text> and <div> elements will typically start with a <head> element. In the case of <text> this is the title of the article. In the case of <div> it is the heading for the section. The <head> element may contain basic inline elements (see below). While the <head> element is optional, in a long article it is good practice to include headings for the divisions. Headings should not consist simply of a number, but should give the reader some idea of where the argument is going.

<epigraph>

The  <epigraph> element is used to encode short quoted passages at the start of a division. Epigraphs may appear at the top of a <text> or <div>, following the <head>. Epigraphs will typically contain a <cit> element, which consists of a quotation (encoded with <quote>) followed by a citation (typically encoded with <ptr> or <ref>, but see below for special cases). A graphic or example may be used in place of <cit>. 

<epigraph>
   <cit>
      <quote rend="block"></quote>
      <bibl></bibl>
   </cit>
</epigraph>

Poetry and Drama

<lg>

Verse is encoded using the <lg> element, which represents a "line group" of one or more poetic lines.

<l>

Within <lg>, individual lines of poetry are encoded with <l>. A "line" in this context does not make significant semantic claims to being verse, but simply represents a line of text that operates as a unit that cannot be arbitrarily broken or relineated.

<sp>

Within dramatic dialogue, individual speeches or utterances are encoded using the <sp> element. This element may contain one or more <p>, <lg>, or <stage> elements in any order.

<stage>

The <stage> element is used to represent stage directions.

Phrase-level Elements

<said>

Direct speech or thought should be represented with <said>. However, most cases of direct speech in DHQ articles will also be quotations, and should be encoded simply with <quote> rather than with <quote> and  <said>. The use of <said> should be confined to cases where direct speech is being uttered within the context of the article (e.g. It's as if one were to say "Hello!" without really meaning it...)

<emph>

The <emph> element is used to encode words and phrases that are intended to be rhetorically emphatic. It will typically display in italics.

<foreign>

The <foreign> element is used to encode words and phrases that are in a language other than that of the surrounding text. It carries an @xml:lang attribute which should always be used to indicate what language its contents are in. The values for the @xml:lang attribute are two-letter language codes as follows:

<hi>

The <hi> element is a fall-back element used to apply necessary formatting that is not driven by one of the more semantically distinctive elements provided in the schema. For instance, it might be used to represent superscripts or subscripts, or to highlight a portion of a code sample to bring it to the reader's attention. Its @rend attribute takes the following values:

  • bold
  • italic
  • monospace
  • quotes
  • smcaps
  • subscript
  • superscript

No other values have any processing attached to them at this time.

<name>

The <name> element is used to encode names in the text: typically the names of persons, but also place names and the names of organizations as well. At present this element is not processed in any special way, but in the future we may provide a more detailed rationale for encoding certain types of names (e.g. the names of significant figures in digital humanities) to support analysis. Authors wishing to encode names are free to do so.

<q>

The <q> element is used to represent words or phrases (other than technical terms, quotations and direct speech) that should be presented in quotation marks. It encompasses the following:

  • expressing ironic distance: the "benevolent" motivations in this case were starkly illustrated...
  • drawing attention to specific usages: the "dungeon" in this case is intended metaphorically...
  • identification of words as words: when we consider the word "phthisis" we are immediately struck...

The <q> element should only be used in cases where neither <quote>, <said>, or  <term> is appropriate.

<term>

The <term> element is used for technical terms, which will typically be displayed with italics. In the future this element may serve as the basis for a DHQ glossary. Authors wishing to encode terms are free to do so.

<title>

The <title> element is used to encode the titles of works discussed or cited in a DHQ article. Within the <bibl> element it represents the title of the work being cited. In running prose, it should be used for any title mentioned.

The @rend attribute on title may take one of three values:

  • italic (for books, journals, and other standalone works such as works of e-literature
  • quotes (for journal articles, individual chapters or sections of larger works)
  • none (for conferences, interviews, book series, and other titles that require no formatting)

For foreign-language titles, the <title> element should carry an @xml:lang attribute indicating what language the title is in. If the title is followed by a transliteration (for instance, a Chinese-language title in both roman alphabet and Chinese characters), each title should be enclosed in a separate <title> element.

In addition, for articles that are discussing markup or programming, the following additional elements may be useful.

Elements for Representing Markup and Code Samples

<att>

The <att> element is used to encode the name of an XML attribute. It will typically be formatted in a way that signals this fact: e.g. using the conventional @ prefix.

<val>

The <val> element is used to encode an XML attribute value. It will typically be formatted in a way that signals this fact: e.g. by enclosing it within quotation marks.

<eg>

The <eg> element is used to encode multi-line literal examples, which may be sample code or other material that needs to be presented with its formatting and line breaks intact. If it contains an XML sample, its contents should be escaped, either by containing them inside a CDATA section, or by escaping the individual characters (we prefer the former). The <eg> element is treated as a block-level element for display purposes, so it will be set off from the surrounding text and all internal white space and lineation will be preserved. By default its contents will be displayed in a fixed-width font. Within <eg>, the <hi> element is permitted to allow for highlighting of significant sections, and the <formula> element is permitted to allow for the representation of mathematical expressions. (See for instance article 000526.xml.)

The @lang attribute is required, to designate the language of the code sample, which supports syntax highlighting. We do not provide automated syntax highlighting for all machine languages. If the example is in a language not on our list of supported languages, there are two alternatives:

  • nohighlight: this will produce a monospace font without syntax highlighting
  • code-general: the syntax highlighter will attempt to guess the appropriate formatting

The distinction between examples and quotations may in some cases be hard to draw; in cases where the example is significant for being quoted from some specific piece of code, the <eg> element may be nested inside a <quote> element.

An example of a code sample with a CDATA marked section: 

<eg><![CDATA[First line of the code sample
   Second line of the code sample (indented three spaces)
   Third line of the code sample (indented three spaces)]]>
</eg>

In the output, this code sample would be formatted as follows: 

First line of the code sample
   Second line of the code sample
   Third line of the code sample

<dhq:example>

The <dhq:example> element is used for any kind of structured examples, including sample code and non-code examples, that includes a caption or requires some internal markup beyond <hi>. and do not require exact preservation of white space (for instance, a sample text that will be discussed in the article). This is a DHQ-specific element; it contains an optional <head> followed by the sample text (which may be encoded with <floatingText>, <eg>, or with other structural elements such as <p>, <sp>, etc. as needed) and an optional <dhq:caption>.

<code>

The <code> element is used for short snippets of code (such as XML or programming languages), typically no more than a single line. It is treated as an inline element for display purposes, so it will wrap with the text that surrounds it. By default, its contents will typically be displayed in a fixed-width font.

<gi>

The <gi> element is used to encode the name of an XML element (without attributes). It will typically be formatted in a way that signals this fact: e.g. by enclosing it within angle brackets. For XML tags that include attributes, use <code> (and include the angle-bracket delimiters since those will not be supplied automatically).

Quotation and Citation

<quote>

The <quote> element is used for quotations of material from outside the text (e.g. other articles, books, aphorisms, etc.).

  • The @rend attribute takes values "inline" and "block"; inline quotations will be formatted with quotation marks; block quotations will be set off as blocks.
  • The @source attribute carries a pointer to the associated <bibl> in the bibliography.
  • When accompanied by a bibliographic citation (encoded as <ptr> or <ref>), a quote should be enclosed within a <cit> element to associate the quote and the citation.

<cit>

The <cit> element is used to represent complete citation: a quotation and the source from which it is taken. It groups together a <quote> element and some form of bibliographic reference.

  • In simple cases, the bibliographic reference should be encoded with <ptr>
  • If the wording of the reference is distinctive and needs to be preserved, it should be encoded with <ref>.
  • If the wording of the reference includes material that should not be included in the link, it should be encoded with <dhq:citRef> (see below), which contains a <ref> or <ptr> plus text content.

In all cases, the @target attribute of <ref> or <ptr> points to the @xml:id of a <bibl> element in the <listBibl> at the end of the document. In rare cases, a <ref> may simply contain a brief citation without pointing to a <bibl> (for instance, if the quotation is from a famous speech or other non-bibliographic source), in which case <ref> should carry a @type attribute with the value "offline".

<dhq:citRef>

The <dhq:citRef> element is used to handle cases where there is material in the citation that is not appropriate to include within <ref> (since the contents of <ref> are a link to an item in the article's bibliography). The <dhq:citRef> element contains a <ref> or <ptr> element, plus whatever text content is present: 

<cit>
<quote rend="block">the physical world is generally modelled as
a series of discrete locations known as <term>rooms</term>. The totality of rooms in a
given work of IF is often referred to as the <term>map</term>. Such rooms could
correspond to rooms in a building, but they need not and frequently do not[...].
Conceptually, a room is that segment of physical space that is immediately accessible
to the player character.</quote>
<dhq:citRef><ptr target="#eve2007" loc="para. 7"/>(emphasis in original)</dhq:citRef>
</cit>

<cit>
<quote rend="inline">Only once in my life have I seen a wonderful medium effectively wiped
out by new technology</quote>

</cit>

<bibl>

The <bibl> element is used to encode individual items in the bibliography for the article. In the future, most bibliographic items will be stored in a comprehensive bibliographic database and their data will be extracted and imported automatically into DHQ articles. However, all items will be represented by a "stub" <bibl> element, typically carrying an attribute that will point to the full bibliographic record in the database. Some items may be unsuitable for inclusion in Biblio, either because they are too odd or because they are too article-specific (e.g. "Personal interview with the author, June 2008"). These should receive a full bibliographic entry in the article itself.

The <bibl> element carries three attributes:

  • @xml:id, which provides a unique identifier so that references in the text can point to their bibliographic item
  • @label, which provides a string that can be automatically printed in the text as a reference (e.g. "Liu 2008")
  • @key, whose value is used as a lookup/link to the Biblio database.

The value of @xml:id should take the form lastnameYYYY, where YYYY is the four-digit year of publication. If there is more than one item by the same author in the same year, then letters may be used for disambiguation: smith2008a, smith2008b, etc. ID values should always be all lower case and should not contain diacritics or punctuation. For multi-author items, use the last name of the first author only.

The value of the @label attribute should be the label that will be displayed in the text when the item is cited, e.g. Smith 2008. The label should be the last name(s) of the authors plus the four-digit year of publication, separated by a space, without a comma. For example:

  • Smith 2008
  • Smith and Jones 2008
  • Smith et al. 2008
  • Sinclair-Smith 2008

The value of @key is typically the same as @xml:id, unless the disambiguation characters within the DHQ article are different from those used within Biblio.

<ptr>

The <ptr> element is an empty element, used for internal bibliographic references and certain external URIs. Its @loc attribute can be used to provide a page number or page range (for printed sources) or a section or paragraph number (for online sources).

The <ptr> element is used in three cases:

  1. For simple internal references to bibliographic items, in cases where the link text can be automatically generated from the @label attribute on <bibl> and the @loc attribute on <ptr> (e.g. "Smith 2008, 19"). The @target attribute on <ref> contains the @xml:id of the targeted bibl, preceded by a hash mark (#): e.g. <ptr target="#smith2008"/>. In cases where the author wants to supply some additional text for the link (e.g. "see Smith's excellent introduction, especially page 19") the <ref> element should be used instead of <ptr>.
  2. For simple references to external URIs from within the bibliography. In these cases, the @target attribute on ref carries the full URI (including the protocol: "http://%22, etc.).
  3. For references to the DHQ "annex", which is where author-supplied code/components that DHQ does not support are stored. We encode these references as <ptr type="dhq-annex-embed">. The @target attribute is used to point to the relevant code or object to be embedded, which is stored in the annex.

For display purposes, the information encoded with <ptr> will be presented either as a URL (in the case of external targets) or as the formatted label of a bibliographic reference (in the case of internal pointers to the bibliography). If there is a value for @loc, the display will include that information as well; for instance:

<ptr target="jones1999" loc="4"/>would display as [Jones 1999, 4]. <ptr target="http://www.digitalhumanities.org%22/>will be presented as <http://www.digitalhumanities.org>.

For values of @loc that are not simple page numbers, a prefix should be included that indicates what kind of reference number is being used: for instance, "para. 1" or "section 3.2" or "item 6". The value of @loc will be displayed exactly as encoded.

<ref>

The <ref> element is very similar to <ptr>, but it must contain content. It is used for several purposes:

  1. cases where the wording of a bibliographic reference is significant and needs to be preserved. For instance, "McGann 2006, 16 (emphasis in original)". In these cases, page number information needs to be represented in the content of <ref> as well as on @loc, so that it is accessible to readers and for processing.
  2. references to external URIs in the body of the article. In some of these cases, the URI should appear both in the content and in the value of @target. In other cases, it may be more appropriate to put the name of the site or page instead of a URI in the content of <ref>; see stylistic note below.
  3. internal cross-references to other sections, notes, etc. in the DHQ article or in other DHQ articles. In these cases, the @target attribute should contain the @xml:id of the element being pointed to (e.g. <ref target="#figure01">Figure 1, below</ref>).

The ref element also carries a @loc attribute; although this is not used for display, it can be used for analysis and to ensure that the page reference is captured in a formal manner.

For example: <ref target="jones1999" loc="note 14">See Jones 1999, note 14</ref> <ref target="http://www.wikipedia.org%22>Wikipedia</ref>

For references to external URIs, <ref> works just like an HTML link; the author provides the text that will be inserted into the anchor of the resulting hyperlink. DHQ house style stipulates that where the URL is unfamiliar or difficult to discover, or where the reference is to a very specific page on a site, the URL should be included as the content of <ref> so that it remains visible if the reader prints the DHQ page; for URLs that are very familiar or easy to discover (e.g. the New York Times, major funding agencies, etc.) the name of the site be used instead to reduce clutter on the page.

The <ref> element may also be used without a target at all. In this case, no link will be generated. This is appropriate for references that are not to specific published items but to works that have conventionalized citation systems (e.g. Homer and most classical texts, the Bible and other scriptural texts), or that are not available in published form at all (e.g. private communications to the author, keynote lectures, etc.). In these cases, <ref> is used with a @type attribute whose value is "offline", indicating that there is neither a bibliographic citation nor a URL available.

Where a citation accompanies an epigraph, it may be desirable for aesthetic reasons to give the bibliographic information in a more extended form (i.e. not the abbreviated "Emerson 1971" but the more extensive "Emerson, The American Scholar". In these cases, the desired display text should be encoded as the content of <ref>, and the target attribute should point to the <bibl> element in the usual way.

Figures, Tables, Media

<figure>

Figures are encoded with the <figure> element. It contains several child elements:

  • an optional <head> containing whatever text the author has supplied to describe the figure (a heading with the figure number will be automatically generated and any author-supplied heading will be appended to this)
  • a required <figDesc> containing a brief description of the figure, to be used when the figure itself cannot be viewed
  • a required <graphic> element contains a @url attribute which points to the relevant image file. The file path is <graphic url="resources/images/figure01.png"/>

Each <figure> element should carry a unique identifier (@xml:id) to permit cross-referencing. The value of the identifier should be "figure01", "figure02", etc. for ease of proofreading.

Image files should be named figure01.jpg (etc.).

<figDesc>

The <figDesc> element contains a short description of the accompanying figure, to be used when the image itself cannot be viewed or displayed. DHQ typically supplies the contents of <figDesc> but in cases where specialized knowledge is required the author should supply a description.

<graphic>

The <graphic> element is an empty element; it carries a @url attribute that points to the relevant image file. It should use a relative path and point to the resources directory for the article, as in the example below. To control the size of the image, use the @style attribute and include a CSS height property with a value that gives the number of pixels, as in the example below. 

<figure>
<graphic url="resources/images/figure01.jpg" style="height: 100px;"/>
</figure>

<dhq:caption>

The <dhq:caption> element is now deprecated in favor of <head>. The following information is provided for historical purposes only. <dhq:caption> was a DHQ-specific element used to encode captions (of figures, tables, and examples). It may contain plain text with phrase-level elements, or a series of paragraphs. The content is supplied by the author; if none is supplied, the element is not needed. This element was only used for captions that contain significant internal structure (such as paragraphing).

<media>

Video materials are hosted on DHQ's Vimeo channel (preferred) or on YouTube. In the article, a <figure> element is used with a <media> element pointing to the Vimeo link. The height and width of the video are specified to avoid excess white space, using @height and @width on the <media> element: 

<figure>
   <head>Locating Grid Technologies: Performativity, Place, Space: Challenging the Institutionalized Spaces of e-Science (2cam2). This camera points towards the rear of the Physics INSORS grid node. The camera is placed to the right of the room space.</head>
   <figDesc>A video showing a dance performance</figDesc>
   <media mimeType="video/mp4" url="https://player.vimeo.com/video/9561872" width="640" height="360"/>
</figure>

If the video is hosted on YouTube, the @height and @width are not necessary. In some situations, the author may want to embed a video in a place where the heading is not desirable. In that case, the <media> element can be used on its own. This should be treated as an edge case (since the <figDesc> potentially provides important accessibility information).

<dhq:passThroughCode>

In situations where the author includes HTML code (or potentially other code) that needs to be executed (and hence needs to be embedded in the TEI markup and passed through for execution to the HTML output) DHQ has created an element <dhq:passThroughCode>. This element can contain any well-formed fragment of XHTML code. This element can be treated as analogous to <dhq:example> and can go in the same kinds of places: either within or between paragraphs. However, if it needs a DHQ caption/heading it can also go inside a <dhq:example> element, with a <head> as the first child. The @lang attribute (not @xml:lang) indicates what formal language, such as XHTML, the code in question is expressed in. The @rend attribute indicates whether the code sample should be treated as an inline or block element. Typical uses for this element include:

  • embedded iFrames (for instance, to embed maps, visualizations, examples from other sites)
  • embedded interactive forms that communicate with other online resources (as illustrated in the example below)

The <dhq:passThroughCode> element should 'not' be used to embed videos hosted on YouTube or Vimeo; those should be encoded with <media>.

For all embedded iFrames, the following code should be included as the content of the iFrame element: 

<p>Your browser does not support iframes. Embedded document may be 
   found at <a href="[link to iframe source]">[link to iframe source]</a>
</p>

<dhq:passThroughCode lang="xhtml" rend="block">
  <div xmlns="http://www.w3.org/1999/xhtml">
   <form action="http://romanrepublic.ac.uk/rdf/repositories/dprr/query" target="_blank">
     <fieldset>
       <textarea id="query" name="query" rows="16" cols="80" wrap="soft">
         PREFIX vocab: <http://romanrepublic.ac.uk/rdf/ontology#>
                select ?name ?officeName 
                where { 
                  ?person a vocab:Person;
                    vocab:hasName ?name;
                    vocab:isSex <http://romanrepublic.ac.uk/rdf/entity/Sex/Female>.

                     vocab:isAboutPerson ?person;
                     vocab:hasOffice ?office.
                  ?office vocab:hasName ?officeName
} </textarea>
        <input type="hidden" name="ref" value="text"/>
        <input type="hidden" name="queryLn" value="SPARQL"/>
        <input type="hidden" name="limit_query" value="0"/>
        <br/>
        <input id="exec" type="submit" value="Execute"/>
      </fieldset>
    </form>
  </div>
  </dhq:passThroughCode>


<list>, <item>

The <list> element is used to encode lists of all kinds. It contains one or more <item> elements. Its @type attribute takes the following values:

  • "ordered": generates numbered labels
  • "unordered": generates bullet labels
  • "gloss": formats the contents of label
  • "simple": no labels at all

<table>, <row>, <cell>

The <table> element is used to encode tables. It contains a series of <row> elements, each of which contains one or more <cell> elements. It may also contain an optional <head>. The <head> element must be inserted as the first child of <table> in order for it to validate correctly (see below for example).  The optional <dhq:caption> is currently deprecated and will not display. 

<table>
   <head>Table Name/Description</head>
      <row>
         <cell></cell>
      </row>
</table>

To identify a given row as a header row (or to identify a given cell as a label for the row it appears in), use the @role attribute, with the value "label": 

<row role="label">
   <cell role="label">
<row>

To enable a cell to span across multiple columns, use the @cols attribute and the number of columns to be spanned. This will reduce the number of <cell> elements in that row; so if a table has (for example) six columns and one of them needs to span three columns, the enclosing <row> should have only four <cell> elements. 

<row>
   <cell cols="6">
</row>


Rows and cells identified as "label" will be formatted in bold type.

The <p> element is not permitted inside table cells; however, to force a line break (e.g. to have multiple sub-items within a cell) you can force a line break with the <lb/> element: 

<cell>
Humanities
<lb/>Social Sciences
<lb/>Natural Sciences
</cell>

Encoding audiovisual materials

Audio materials are hosted locally: the audio file itself (MP3 preferred) is stored in the article's resources directory: /resources/audio/filename.mp3.

In the article, a <ref> element is used to point to the audio file, e.g.: 

<p>Click for the accompanying <ref target="resources/audio/john_unsworth.MP3">audio interview</ref>.</p>


Video materials are hosted on DHQ's Vimeo channel (preferred) or on YouTube. In the article, a <figure> element is used with a <media> element pointing to the Vimeo link. The height and width of the video are specified to avoid excess white space, using @height and @width on the <media> element: 

<figure>
   <head>Locating Grid Technologies: Performativity, Place, Space: Challenging the Institutionalized Spaces of e-Science (2cam2). This camera points towards the rear of the Physics INSORS grid node. The camera is placed to the right of the room space.</head>
   <figDesc>A video showing a dance performance</figDesc>
   <media mimeType="video/mp4" url="https://player.vimeo.com/video/9561872" width="640" height="360"/>
</figure>

If the video is hosted on YouTube, the @height and @width are not necessary. In some situations, the author may want to embed a video in a place where the heading is not desirable. In that case, the <media> element can be used on its own. This should be treated as an edge case (since the <figDesc> potentially provides important accessibility information).

Mathematical Expressions

We encode mathematical expressions using TeX or MathML. Knowledgeable authors may submit articles that use MathML or TeX, in which case we can use that code directly.

Mathematical expressions should be encoded within the <formula> element. The notation attribute indicates which notation is being used:

  • notation="asciimath" [deprecated]
  • notation="tex"
  • notation="mathml"

The rend attribute indicates whether the mathematical expression is to be rendered inline (flowed in with the rest of the text) or as a separate block with whitespace surrounding it:

  • rend="inline"
  • rend="block"

ASCIIMath (deprecated for the time being)

For expressions encoded with ASCIIMath, each expression must begin and end with the ` character. This delimiter is a signal to the JavaScript that processes the display, letting it know which parts of the code are ASCIIMath.

Example: <formula notation="asciimath" rend="inline">`ax^2 + bx + c = 0`</formula>

ASCIIMath is currently deprecated for DHQ, pending an update to the ASCIIMath code that is included in MathJax.

TeX

For expressions encoded with TeX, inline expressions must begin with \( and end with \) (i.e. escaped parentheses). Block-level expressions must begin and end with a double dollar sign ($$). These delimiters are a signal to the JavaScript that processes the display, letting it know which parts of the code are TeX.

Examples:

<formula notation="tex" rend="inline">\(ax^2 + bx + c = 0\)</formula>

<formula notation="tex" rend="block">$$ax^2 + bx + c = 0$$</formula>

MathML

For expressions encoded with MathML, no special delimiters are needed, but within the <formula> element the MathML encoding must be enclosed within a <m:math> element.

Example:

<formula notation="mathml" rend="inline"> 

<m:math> <m:mi>a</m:mi><m:mo>=</m:mo><m:mn>0</m:mn></m:math>

</formula>

To test MathML encoding, paste just the MathML portion into one of these sites:

https://jsbin.com/riducokufa/1/edit

https://www.ncbi.nlm.nih.gov/pmc/utils/mathtool/

and visually compare the output to the author's final submitted version. A few pointers to remember:

  • The MathML namespace needs to be declared on the root element
  • The MathML encoding itself needs to EITHER have the MathML namespace on every MathML element, or needs to have a prefix declared (and used on every MathML element), or needs to declare the namespace on every >math< element (in which case the prefix is not needed for the child elements).
